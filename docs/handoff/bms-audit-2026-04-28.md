After clicking through every BMS surface in Gulino's tenant as super_admin and as agent (Anthony Ferraioli), the demo went poorly for **three reasons**, in priority order:

**1. Financial numbers are inconsistent across surfaces.** The same brokerage shows four different "paid out" totals: `$0` on Dashboard, `$36,179` on Deal Submissions, "18 in Payment Received" on Transactions, `$0` on Payments. Root cause: the codebase has **three parallel deal stores** — CRM `Deal`, BMS `Transaction`, and `DealSubmission` — and different surfaces read from different tables. The historical Gulino import populated `DealSubmission` but didn't backfill `Transaction` / `Invoice` / `Payment` rows. Manager can't trust any number on the dashboard.

**2. The super_admin `?as_org=` override is inconsistently scoped.** It works for queries against `DealSubmission` / `Transaction` / `Invoice` but **falls through to the home-org for queries against `User` / `BrokerAgent` and for routes that redirect** (Reports drops the param, Agent Roster shows Nathan's agents, Settings → Role Assignment shows Nathan's users). Phase D + the client-side sweep covered the major surfaces but missed the user-table queries.

**3. Information architecture is investor-first, not brokerage-first.** Top-level sidebar order: Dashboard → Research → Portfolio → **Acquisitions** → **Closing** → Leasing → Screening → Property Mgmt → **Brokerage**. For John & Kristin, Brokerage is buried at the bottom; for agents, "Acquisitions" and "Closing" are jargon they don't use. Inside Brokerage, 14 items split into 7 categories (one of which contains a single item). Two sidebars compete for ~30% of the viewport. Mixed icons (lucide + emoji). No role-specific home page.

**The agent surface is meaningfully better than the manager surface.** `/brokerage/my-deals` uses card UI, payout transparency ("after 2.00% fee" callout in red), stage progress dots, and a clear "Submit a Deal" CTA. The manager dashboard has none of these — just dense KPI grids and table lists.

**Recommendation:** ship a 5-slice redesign over ~4 weeks that (a) consolidates deal data into a single source of truth, (b) hardens the super_admin override, and (c) builds **role-specific home pages** with primary-action UIs. Don't rework the whole sidebar at once — do it surface-by-surface starting with the deal-submissions approval queue (manager) and deal-submission form (agent). Detailed phasing in the **Redesign proposal** section below.

**18 bugs** logged (3 P0, 6 P1, 9 P2/P3). **38 UX issues** logged. Full inventory + per-surface findings below.

## Redesign proposal

### Guiding principles

1. **One source of truth per concept.** Deals, invoices, and payments should each have a single canonical store. Surfaces read from the canonical store, not parallel ones.
2. **Role-first IA.** Brokerage admin sees a brokerage-shaped app. Agent sees an agent-shaped app. Super_admin sees both with tenant override controls. Don't make John & Kristin navigate around investor / researcher destinations.
3. **Primary action per page.** Every screen has one big primary action and ≤2 secondary actions. No more 4-CTA headers (Invoices currently violates this).
4. **Defer power-user features.** Bulk import, advanced filters, custom commission tiers — keep them, but don't put them in the primary nav. New tenants should be able to walk the happy path without seeing them.
5. **Empty states teach.** When a brokerage has 0 invoices, the screen tells them how to make one. (Payments already does this; replicate everywhere.)
6. **Ship in slices, not big-bang.** Each slice fixes one workflow end-to-end and is independently demoable to John or Kristin.

### Phase 0 — Data layer + override consistency (Week 1, parallel to UX work)

This is **engineering, not design.** Without it, every UX fix lands on broken data.

- **Slice 0a — Single deal data model.** Decide: is `DealSubmission` the canonical record? `Transaction`? `Deal`? Pick one. Migrate the others into views or remove. (Recommend: keep `DealSubmission` for inbound + `Transaction` for closed deals, deprecate CRM `Deal` for BMS use cases. Backfill `Transaction` rows for any historical paid `DealSubmission`.)
- **Slice 0b — Backfill Gulino's missing invoices + payments.** 18 paid `DealSubmission` rows, 0 `Invoice` rows, 0 `Payment` rows. Either backfill records OR refactor surfaces to derive from submissions. (Backfill is less risky; matches the "Paid" status the data already claims.)
- **Slice 0c — Harden `?as_org=` override.** Pull every db query that takes `orgId` through one helper (`getCurrentOrgContext()` already exists per memory). Sweep `User`, `BrokerAgent`, `CommissionPlan`, all `Reports`, and any `redirect()` paths. Add a unit test per surface: "as super_admin with `?as_org=X`, surface S queries for org X, not Y."
- **Slice 0d — Fix banner cut-off (B-001).** Trivial CSS / z-index fix.

### Phase 1 — Manager approval queue + invoice flow (Week 2)

The manager's most-trafficked workflow: **review submission → approve → push to invoice → record payment.** This is what John & Kristin do daily. Today it's spread across 5 separate pages with mismatched data.

- **Slice 1 — Unified "Pending Approval" queue.** New page `/brokerage/inbox` (or rename `/brokerage/deal-submissions`). Single screen lists pending submissions with inline expand-to-detail. Approve action is one click; "Approve & Push to Invoice" is the primary CTA. Replaces the current dense table with a card layout (steal patterns from agent's My Deals).
  - Drop: 4 KPI cards at top, 6 status tabs, 4-dropdown filter row.
  - Keep: search.
  - Add: "Pending review" count badge in sidebar.
- **Slice 2 — Invoice creation in-context.** When manager approves a submission, an Invoice draft auto-creates. Manager doesn't navigate to `/brokerage/invoices` separately. Invoice tab on the deal detail panel.
- **Slice 3 — Payment recording in-context.** Same — record payment on the deal, not on a separate page.
- **Result after Phase 1:** Submission, Invoice, Payment are one workflow. Sidebar items "Submissions / Invoices / Payments" can stay (for inspection) but are no longer how the manager moves between steps.

### Phase 2 — Manager dashboard rebuild (Week 2, parallel)

- **Slice 4 — Role-specific dashboard.** Replace `/brokerage/dashboard` with two separate landings: `/brokerage` (manager) and `/brokerage/my` (agent). Manager dashboard shows: pending review count (linked), today's tasks, agent leaderboard top 3, this-month financials (4 KPIs max).
- **Slice 5 — Hide screening KPIs from BMS dashboard.** Move to Screening dashboard. (U-010.)
- **Slice 6 — Default landing per role.** Manager → Brokerage Dashboard. Agent → My Deals. Super_admin → admin home (or last-visited tenant). (B-018.)

### Phase 3 — Sidebar IA overhaul (Week 3)

- **Slice 7 — Single sidebar per role.** Stop showing global sidebar + brokerage sub-sidebar simultaneously. For brokerage admins, collapse global sidebar to icon-only. For agents, hide it entirely (their "MY WORK / COMMUNICATION / RESEARCH" is sufficient).
- **Slice 8 — Brokerage nav flatten.** From 7 sections × 14 items to 3 sections × 8-10 items. Suggested groupings:
  - **Daily** — Inbox (formerly Submissions), Invoices, Payments, Agents
  - **Reports** — Dashboard, Leaderboard, Reports (P&L, 1099, Production)
  - **Setup** — Listings, Properties, Commission Plans, Compliance, Onboarding Templates, Settings
  - Move Bulk Invoices into Invoices (as overflow action). Move "My Deals" out of Brokerage entirely (agent-only top-level).
- **Slice 9 — Replace ALL CAPS / emoji icons.** Standardize on lucide. Mixed-case section labels.
