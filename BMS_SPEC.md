# VettdRE ‚Äî Brokerage Management System (BMS) Spec

## Vision
The BMS is a standalone toolset inside VettdRE that gives brokers everything they need to operate a brokerage ‚Äî from deal intake to commission payouts. Designed to eventually be extracted as a white-label BaaS product.

Flow: SUBMIT ‚Üí APPROVE ‚Üí INVOICE ‚Üí PAY ‚Üí REPORT

## Navigation
Top-level sidebar item "Brokerage" (owner/admin) opens `/brokerage/dashboard`.

Brokerage uses a **vertical secondary sidebar** (matching the Settings layout pattern) with grouped sections:

**OVERVIEW** ‚Äî Dashboard
**INVENTORY** ‚Äî Listings, Properties
**DEALS** ‚Äî Submissions, Invoices, Bulk Invoices, Transactions
**MANAGEMENT** ‚Äî Agents, Commission Plans, Compliance
**FINANCE** ‚Äî Payments, Reports
**ADMIN** ‚Äî Settings

Mobile: horizontal scrolling pills (same as Settings mobile).
Role-gated: each link checks `canAccessPage(role, href)` from `bms-permissions.ts`.

Separate sidebar item "My Deals" (agent role) ‚Üí `/brokerage/my-deals`

Includes CSV template download.

## Database Models

### BrokerAgent
Linked to Organization (orgId) and optionally to User (userId). Fields: firstName, lastName, email, phone, licenseN, defaultSplitPct, status. Table: broker_agents.

### DealSubmission
Linked to Organization and optionally BrokerAgent. Stores all form fields as snapshots. Uses BmsDealType enum (sale/lease/rental). Status is a string: submitted, under_review, approved, invoiced, paid, rejected. Has optional one-to-one Invoice relation. Table: deal_submissions.

### Invoice
Linked to Organization, optionally to DealSubmission (one-to-one) and BrokerAgent. All brokerage, agent, and deal info snapshotted at creation time. Uses InvoiceStatus enum (draft/sent/paid/void). Auto-generated invoiceNumber (INV-YYYY-NNNN). Table: invoices.

### CommissionPlan
Linked to Organization. Fields: name, type (CommissionPlanType: flat/volume_based/value_based), isDefault, isActive. Has many CommissionTier and many BrokerAgent. Table: commission_plans.

### CommissionTier
Linked to CommissionPlan. Fields: tierOrder, minValue, maxValue (nullable for open-ended), agentSplitPct, houseSplitPct. Table: commission_tiers.

### Organization additions
- submissionToken: unique string for public submission links

## Server Action Patterns
- Auth: getCurrentOrg() via authProviderId ‚Üí orgId (same as all other actions)
- Public submission: looks up org by submissionToken, no auth
- All queries scoped to orgId
- JSON.parse(JSON.stringify()) for serialization
- Graceful error handling, never crash

### Bulk Invoice Server Actions (bulk-actions.ts)
- `getBillToMappings()` ‚Äî load saved Bill To entities from bmsSettings
- `saveBillToMappings(mappings)` ‚Äî persist Bill To entity mappings
- `getNextInvoiceNumber()` ‚Äî auto-increment from last invoice in DB
- `getFromInfo()` ‚Äî brokerage name/address/phone/email from org + brand settings
- `getInvoiceSettings()` ‚Äî invoice prefix, notes, line format, payment terms
- `getPaymentInstructions()` ‚Äî payment instructions from bmsSettings (ACH/Wire/Check)
- `getBrokerageLogo()` ‚Äî logo URL from brand settings or org
- `lookupAgentsByName(names)` ‚Äî fuzzy match agent names to BrokerAgent records, returns license numbers
- BrokerAgent matched by email on submission creation

## Phase 5: Transaction Management

### Overview
Pipeline-style transaction tracker for rentals and sales. Each transaction follows a stage-based workflow with checklist tasks that must be completed before advancing stages.

### Database Models (4 new)
| Model | Purpose |
|-------|---------|
| **Transaction** | Core record ‚Äî type (rental/sale), stage, property info, deal amounts, two parties, 10 milestone dates, cancel tracking |
| **TransactionTask** | Checklist items per transaction ‚Äî linked to a stage, sortOrder, isRequired flag, completion tracking |
| **TransactionTemplate** | Org-scoped task templates (one default per type) |
| **TransactionTemplateTask** | Template task definitions |

### Enums (2 new)
- **TransactionStage** ‚Äî `application`, `approved`, `lease_signing`, `move_in`, `offer_accepted`, `under_contract`, `due_diligence`, `closing`, `closed`, `cancelled`
- **BmsTransactionType** ‚Äî `rental`, `sale` (separate from CRM `TransactionType` which includes `new_development`)

### Stage Sequences
**Rental:** Application ‚Üí Approved ‚Üí Lease Signing ‚Üí Move-In ‚Üí Closed
**Sale:** Offer Accepted ‚Üí Under Contract ‚Üí Due Diligence ‚Üí Closing ‚Üí Closed

### Default Checklist Tasks
**Rental (13 tasks):** Application received, Credit check, Income verification, References, Landlord approval, Lease drafted, Lease reviewed, Lease signed, Security deposit, First month's rent, Keys delivered, Move-in inspection, Commission invoice sent

**Sale (15 tasks):** Offer accepted, Attorney retained, Contract signed, Title search ordered/cleared, Home inspection scheduled/completed, Appraisal ordered/completed, Mortgage commitment, Survey, Final walkthrough, Closing scheduled/completed, Commission invoice sent

### Server Actions (`brokerage/transactions/actions.ts`)
| Action | Description |
|--------|-------------|
| `createTransaction` | Create with template tasks auto-populated |
| `createTransactionFromSubmission` | Snapshot DealSubmission data, map BmsDealType ‚Üí BmsTransactionType |
| `getTransactions` | List with search (address, name, client, otherParty), type/stage filters |
| `getTransaction` | Detail with tasks (sorted), agent, dealSubmission |
| `updateTransaction` | Update fields with audit logging |
| `advanceStage` | Check required tasks in current stage, return error with incomplete list if any |
| `revertStage` | Move back one stage, clear close timestamps if reverting from closed |
| `cancelTransaction` | Set cancelled stage + timestamps |
| `toggleTask` | Toggle completion with completedAt/completedBy |
| `addTask` / `updateTask` / `deleteTask` | Custom task CRUD |
| `reorderTasks` | Bulk sortOrder update |
| `getTemplates` / `ensureDefaultTemplates` | Template management with lazy seeding |
| `getTransactionStats` | Total, byStage, byType, avgDaysToClose, openCount, closedThisMonth |

## Phase 9A: Agent Leaderboard & Goals (Data Model + Engine) ‚úÖ

### Overview
Goal-setting and leaderboard system for tracking agent performance. Brokers set monthly targets; the system aggregates real-time data from transactions, deal submissions, and listings to rank agents by target completion.

### Database Models (2 new)
| Model | Purpose |
|-------|---------|
| **AgentGoal** | Per-agent monthly targets + denormalized actuals. Fields: year, month, 4 target/actual pairs (deals closed, revenue, listings leased, listings added), customTargets JSON, lastCalculatedAt. Unique on [orgId, agentId, year, month]. |
| **AgentBadge** | Earned badges/achievements. Fields: type, name, description, earnedAt, period. Types include lifetime (first_deal, deals_10, deals_50, century_club), streak (streak_3, streak_6, streak_12), monthly (monthly_mvp, fastest_lease, top_revenue). |

### Badge Definitions (`src/lib/agent-badges.ts`)
- 10 badge types across 3 categories: lifetime, streak, monthly
- Lifetime: First Deal, Double Digits (10), Half Century (50), Century Club (100)
- Streak: 3-Month, Half-Year, Iron Streak (12 months)
- Monthly: MVP, Speed Demon, Money Maker

### Server Actions (`brokerage/leaderboard/actions.ts`)
| Action | Description |
|--------|-------------|
| `setAgentGoals` | Upsert targets for agent/year/month (admin/broker only) |
| `setBulkGoals` | Set same targets for all active agents |
| `getAgentGoals` | Get goals for specific agent/period |
| `getGoalsByMonth` | Get all agents' goals for a month |
| `calculateAgentActuals` | Aggregate from Transactions + DealSubmissions + BmsListings, deduplicate, update goal record |
| `refreshLeaderboard` | Recalculate all agents for a month + check badges |
| `getLeaderboard` | Ranked agents with scores, streaks, badges, trends. Supports: single month, current_quarter, current_year |
| `getAgentDashboard` | Personal stats: goals, rank, streak, badges, 12-month history, lifetime stats |
| `calculateStreak` | Count consecutive months hitting all targets (backward from current) |
| `checkAndAwardBadges` | Idempotent badge award check (lifetime + streak) |
| `awardMonthlyBadges` | End-of-month MVP + top revenue badges |

### Scoring System
- Weighted target completion: deals (3x), revenue (2x), listings leased (2x), listings added (1x)
- Cap at 150% to prevent runaway scores
- Agents ranked by how well they hit THEIR targets (not absolute numbers)
- Score range: 0-150

### Permissions (3 new keys)
| Key | Roles |
|-----|-------|
| `goals_view` | brokerage_admin, broker, manager, agent (agents see own only) |
| `goals_manage` | brokerage_admin, broker |
| `leaderboard_view` | brokerage_admin, broker, manager, agent |

### File Additions

```
  agent-badges.ts         ‚Äî Badge definitions, lifetime/streak thresholds, AgentLifetimeStats interface
```

### Key Design Decisions
- Actuals are denormalized on AgentGoal for fast reads (lazy refresh on page load)
- Deduplication: if DealSubmission has linked Transaction, count only Transaction
- Revenue priority: Transaction.commissionAmount > DealSubmission.totalCommission
- Streak calculation works backward from previous month (current month in progress)
- Badge checks are idempotent ‚Äî safe to call repeatedly
- No UI in this phase ‚Äî data model and engine only
